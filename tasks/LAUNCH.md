# LAUNCH — как запускать агентов по задачам

## Принципы

1. **Одна задача = одна сессия агента = одна ветка = один PR.** Не давайте агенту две
   задачи в одной сессии — контекст загрязняется, слабая модель начинает смешивать scope.
2. **Сначала T14 (CI).** После него каждый PR автоматически проверяется тестами и
   сборкой — это ваш фильтр качества для всех остальных задач.
3. **Задачи выполняются последовательно**, по одной. Параллельно можно запускать только
   задачи одной волны и только в отдельных git worktree — но для соло-разработки
   последовательный режим проще и безопаснее.
4. **Стек должен быть поднят** перед запуском агента: `docker compose up -d` — иначе
   агент не сможет прогнать проверки.
5. Агент работает по циклу из CLAUDE.md: план → ваш OK → код → проверки → предложение
   коммита → ваш OK. Не отключайте это — на слабой модели подтверждение плана ловит
   ошибки понимания до того, как они попадут в код.

## Запуск сессии

```bash
cd /Users/Ilya/VS-Code/gps-heatmap
docker compose up -d
git checkout main && git pull && git checkout -b task/t01-tracks-pagination
claude --model sonnet        # или haiku; модель также меняется через /model
```

Дальше вставьте промпт из шаблона ниже.

## Шаблон промпта (подставить номер задачи)

```
Выполни задачу tasks/T01-tracks-api-pagination.md.

Порядок работы:
0. Подними стек, если не поднят: docker compose up -d (проверь docker compose ps).
   Убедись, что ты на актуальном main (git status, git pull), затем создай
   ветку задачи и работай в ней.
1. Прочитай ЦЕЛИКОМ: CLAUDE.md, tasks/README.md (раздел «Правила для агента»
   и «Инварианты проекта») и файл задачи.
2. Прочитай разделы architecture/ARCHITECTURE.md, указанные в задаче, и каждый
   файл, который собираешься менять — прежде чем менять.
3. Напиши короткий план изменений (файлы, что именно меняешь) и жди моего OK.
4. После OK реализуй строго по разделу «Что сделать». Раздел «Чего НЕ делать» —
   жёсткое ограничение: если кажется, что без нарушения не обойтись, остановись
   и спроси меня.
5. Прогони ВСЕ команды из раздела «Как проверить». Красные тесты/сборку чини до
   зелёного состояния. Вывод проверок покажи мне.
6. Обнови документацию из раздела «Документация».
7. Сверься с «Критериями приёмки» пункт за пунктом и покажи итог: что изменено,
   результат по каждому критерию. Предложи коммит и жди OK.

Правила: npm только через docker compose exec. Dev-сервер УЖЕ запущен (сервис
frontend в docker compose — это npm run dev с HMR): не запускай его ни локально,
ни в контейнере. Работаем в ветке — на main не переключаться. Заметил проблему
вне задачи — запиши одной строкой в POLISH.md, не чини. Секреты в чат не
выводить. НЕ останавливай стек (docker compose down) в конце сессии — он нужен
следующим задачам.
```

## Дополнения к промпту для отдельных задач

Добавьте соответствующую строку в конец шаблона:

- **T14:** «После моего OK на коммит: закоммить, запушь ветку, открой PR через gh
  и добейся зелёного прогона workflow (gh run watch). Если CI падает — чини и пушь
  снова, до зелёного.»
- **T04, T05:** «Перед планом обязательно изучи, как связаны App.jsx, appStore,
  mapStore и слои карты (map/*.jsx) — в плане опиши текущий поток данных, чтобы я
  проверил твоё понимание.»
- **T11:** «Прод-компоуз проверяй локально (docker compose -f docker-compose.prod.yml).
  На реальный VDS ничего не деплой.»
- **T13:** «Ничего не выдумывай: каждое утверждение в доках сверяй с кодом. Не уверен —
  спроси меня, а не пиши приблизительно.»

## Цикл после каждой задачи

1. Агент показал итог → просмотрите diff (`git diff main`) и критерии приёмки.
2. OK на коммит → агент коммитит → push ветки → PR → дождаться зелёного CI (после T14).
3. Merge в main, `git checkout main && git pull`, удалить ветку.
4. Проставьте ✅ задаче в tasks/README.md (можно попросить это сделать следующего агента).
5. Следующая задача — с чистой сессии.

## Если агент ушёл не туда

- Мелкая ошибка — укажите конкретно («критерий 2 не выполнен: …»), пусть чинит.
- Систематически не туда — не спорьте с сессией: `git checkout . && git clean -fd`
  (сброс незакоммиченного), новая сессия, при необходимости модель выше.
- Замечание к самой формулировке задачи — поправьте файл задачи, потом перезапуск:
  файл задачи — источник правды, а не переписка.
