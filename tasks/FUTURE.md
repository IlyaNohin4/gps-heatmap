# FUTURE — заготовка на рост до многопользовательского сервиса

Проект сейчас — личный инструмент (1-5 пользователей, VDS 2 CPU / 8GB). Пункты ниже
**сознательно не делаем**: на текущем масштабе они не окупаются. Возвращаться к этому
файлу при появлении сигналов из колонки «Триггер».

| Что | Зачем | Триггер («пора делать, когда…») | Оценка |
|-----|-------|--------------------------------|--------|
| **Rate limiting на всё API** (auth уже закрыт в T15) | Защита от абьюза data-endpoints | Открытая регистрация / публичный домен с чужими пользователями | 1-2h |
| **Sentry** (backend + frontend) | Ошибки чужих пользователей не видны в консоли | >5 активных пользователей | 2-3h |
| **CD — автодеплой по тегу** (CI-тесты уже есть, T14) | Ручной деплой станет узким местом | Деплой чаще ~1 раза в неделю или второй контрибьютор | 2-3h |
| **Playwright E2E в CI** | Полный прогон UI-тестов на каждый push | Когда прод-компоуз стабилен и E2E-тесты не флакают | 2h |
| **Full-text search** (pg_trgm GIN-индекс на tracks.name, poi.name) | ILIKE '%…%' медленный на больших объёмах | Поиск заметно тормозит (>100ms) при >50k записей | 2h |
| **Серверный heatmap** (пре-агрегация тайлов вместо L.heatLayer по всем точкам) | Фронтовый heatLayer пересчитывает всё при каждом движении карты | Лаги карты при >2-3k треков; сначала измерить профайлером | 4-6h |
| **Geometries с bbox/зумом** (`GET /tracks/geometries?bbox=&simplify=`) | Один bulk-ответ станет слишком большим | Ответ /geometries > ~10-20MB | 2-3h |
| **Партиционирование tracks** (по user_id или дате) | Таблица перестанет помещаться в индексы | >1M строк в tracks | 3-4h |
| **Разделение appStore на слайсы** (tracks/ui slices) | Поддерживаемость store при росте фич | Store станет >300-400 строк или начнёт путать | 2-3h |
| **Разбиение LeftIsland** (FilterPanel / TracksList / SearchBar) | Компонент растёт | LeftIsland > ~400 строк | 2-3h |
| **Метрики/uptime** (Prometheus или хотя бы uptime-пинг) | Понимать, жив ли сервис | Сервисом пользуется кто-то кроме автора | 2-8h |
| **Секрет-менеджер** (Vault/SOPS вместо .env на VDS) | .env на диске — приемлемо только для малого масштаба | Несколько серверов или команда | 3h |
| **Managed Postgres / миграция с VDS** | 8GB RAM упрётся в потолок | Постоянная нагрузка CPU/RAM > 70% | — |

## Примечания

- **Email (Resend)** — не «рост», а недоделанная фича (сброс пароля): остаётся
  в POLISH.md как продуктовая задача.
- Перед любой оптимизацией карты — сначала профилирование (DevTools Performance),
  как это делалось в perf-сессии 2026-07-05.
