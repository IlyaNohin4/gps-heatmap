# T28 — TCX/FIT экспорт из Track Creator: невалидные файлы

**Приоритет:** P2 · **Оценка:** зависит от решения (см. ниже, вероятно 1-3h) ·
**Зависимости:** нет ·
**Происхождение:** POLISH.md, «Экспорт TCX/FIT некорректен» (архитектурное
ревью, 2026-07-12).

## Уточнение scope (важно, отличается от исходной записи в POLISH.md)

POLISH.md утверждает, что `_points_to_tcx`/`_points_to_fit` используются «и в
обычном экспорте (`/api/tracks/{id}/export`), и в потоке создания трека
вручную на карте». **Это устарело/неверно на момент постановки T28:**
`grep -n "export" backend/app/api/*.py` — эндпоинта `/api/tracks/{id}/export`
в кодовой базе нет вообще (возможно, был убран после ревью или запись была
ошибочной с самого начала). Единственный вызов обеих функций —
`create_track` (`backend/app/api/tracks.py:298-339`), то есть **только
Track Creator** (рисование трека вручную на карте, `body.points: List[Point]`
с `lat`/`lon`).

Также важно: `Point` (`tracks.py:33-35`) содержит **только `lat`/`lon`** — ни
высоты, ни времени у нарисованного трека нет вообще, ни в одном формате.
Даже GPX/KML-экспорт того же `create_track` сейчас пишет `<ele>0</ele>` без
`<time>` (`_points_to_gpx`/`_points_to_kml`, строки 65-85) — то есть
хардкод нуля в TCX это не регрессия относительно других форматов, это
следствие того, что у рисованного трека нет исходных elevation/time данных.

**Из этого следует развилка решений** (см. «Что сделать»):
- либо TCX/FIT для Track Creator принципиально не могут содержать реальные
  высоту/время (их неоткуда взять) — тогда вопрос только в валидности файла
  как контейнера (проходит ли он валидацию сторонним ПО с нулевыми/
  синтетическими значениками), а не в «потере данных»;
  либо стоит убрать TCX/FIT из списка форматов, в которые можно сохранить
  нарисованный трек, если валидность всё равно недостижима без больших
  усилий.

## Текущие проблемы (детали)

- **TCX** (`_points_to_tcx`, `tracks.py:107-124`): каждая точка получает
  `AltitudeMeters=0` и `Time=2024-01-01T00:00:00Z` — фиксированные
  константы, не связанные с реальным временем создания трека.
- **FIT** (`_points_to_fit`, `tracks.py:127-139`): кастомный бинарный формат
  (`.FIT` магический заголовок + `struct.pack('<d', lat/lon)` + заглушка на
  4 байта на точку) без CRC-16 заголовка/трейлера и без обязательных FIT
  message definitions по спецификации Garmin. Результат не является валидным
  `.fit` файлом — сторонние приложения (Strava, Garmin Connect), скорее
  всего, его отклонят.
- Смежная, менее приоритетная деталь: `_points_to_gpx`/`_points_to_kml`/
  `_points_to_tcx` собирают XML вручную через f-строки без экранирования
  (`xml.sax.saxutils.escape()`). Сейчас в шаблоны попадают только числа
  (координаты) — не опасно. Но трек создаётся через `body.name` (строка,
  идёт в БД, не в этот XML) — на сегодняшний день пользовательский текст в
  сам XML не попадает нигде в этих трёх функциях. Уязвимость гипотетическая
  (на случай будущих правок), не эксплуатируется сейчас.

## Что сделать

**Шаг 0 (обязательно первым, до кода):** обсудить с пользователем и
зафиксировать решение по развилке:
1. **Чинить через библиотеки** — `fit-tool` или `garmin-fit-sdk` для валидного
   FIT-бинарника, нормальная сериализация TCX (можно оставить
   `AltitudeMeters=0`/`Time=<текущее UTC на момент экспорта>`, раз реальных
   данных нет, но само содержимое файла должно проходить валидацию по
   спецификации).
2. **Убрать TCX/FIT** из `ALLOWED_FORMATS`, доступных для `create_track`
   (или сузить `ALLOWED_FORMATS` для Track Creator отдельно от форматов
   загрузки файла, если `ALLOWED_FORMATS` используется и там, и там —
   проверить `tracks.py` на предмет одного списка на оба случая).
3. Оставить как есть, но задокументировать ограничение в UI (например,
   предупреждение в Track Creator при выборе TCX/FIT) — если пользователь
   решит, что для личного инструмента реальная валидность не критична.

Дальнейшие шаги зависят от выбранного варианта — детализировать после
решения по шагу 0.

**Отдельно, независимо от варианта:** экранирование в XML-сборке
(`xml.sax.saxutils.escape()` на все текстовые/числовые поля, попадающие в
f-строки `_points_to_gpx`/`_points_to_kml`/`_points_to_tcx`) — небольшая
правка, можно сделать в этой же задаче.

## Чего НЕ делать

- Не трогать `_points_to_gpx`/`_points_to_kml`/`_points_to_geojson` кроме
  экранирования (п. выше) — они не входят в список некорректных форматов.
- Не добавлять elevation/time в Track Creator UI (рисование маршрута с
  профилем высоты) — это отдельная, гораздо более крупная фича, не входит
  в scope T28.
- Не трогать `process_track.py`/`parser_factory.py` — эта задача только про
  экспорт нарисованных треков, не про обработку загруженных.

## Критерии приёмки

Зависят от выбранного варианта (шаг 0). Общее для всех вариантов:
- `docker compose exec backend python -m pytest` — зелёный.
- Если выбран вариант 1 (валидные файлы): экспортированный TCX проходит
  базовую XML-валидацию по TCX XSD (или хотя бы парсится обратно `lxml`),
  FIT проходит через `fit-tool`/аналог обратным парсингом без ошибок.
- Если выбран вариант 2 (убрать форматы): TCX/FIT больше не предлагаются в
  UI Track Creator, `ALLOWED_FORMATS` для этого потока сужен, существующие
  сохранённые треки в этих форматах не затронуты (только создание новых).
