# GPS Heatmap — Правила разработки

## Работа с архитектурой и документацией

1. **Читай сначала архитектуру перед любой задачей**
   - `architecture/INDEX.md` — основная информация
   - `architecture/PARSER.md` — детали парсинга GPS
   - Решай задачи только в рамках принятой архитектуры

2. **Обновляй документацию после решения**
   - Новое архитектурное решение → добавь в соответствующий файл
   - После любого изменения архитектуры → обнови `architecture/INDEX.md` (особенно таблицы зависимостей и контракты)
   - Проблемы при разработке → запиши в `POLISH.md`

3. **Процесс разработки**
   - Получил задачу → прочитал архитектуру → запросил подтверждение плана у пользователя
   - После принятия решения → предложи закоммитить изменения
   - Только пользователь может запросить push в репозиторий

4. **Ограничения**
   - ❌ Никогда не меняй код без подтверждения пользователя
   - ❌ Никогда не пуши в репозиторий без явного запроса
   - ❌ Служебные файлы (.claude, .env, node_modules, __pycache__) не коммитятся
   - ❌ Секреты, ключи, пароли — никогда не выводи в чат
   - ⚠️ При пуше → всегда проверь, нет ли секретов, предупреди пользователя

## Что читать в архитектуре для разных задач

| Задача | Файлы |
|--------|-------|
| Добавить API эндпоинт | INDEX.md (API, Models) |
| Менять парсер | PARSER.md, INDEX.md (Models, Database) |
| Менять UI / дизайн | INDEX.md (Frontend, Islands) |
| Добавить фоновую задачу | INDEX.md (Celery) |
| Менять БД | INDEX.md (Database), PARSER.md (if parse contract changes) |
| Исправлять баги | POLISH.md, затем INDEX.md если нужны архитектурные изменения |
